# 新機能 段階計画スキル

新機能の実装計画を **Plan モード相当の詳細設計** で立て、それを **`/add-feature` コマンド単位の段階リリース計画** に分解し、`docs/ideas/` 配下に **2 つの永続ドキュメント** として保存するスキルです。

## このスキルの目的

1. **大きな機能を段階分割する**: 1 PR で全部やらず、レビュー可能な粒度に分ける
2. **計画を永続化する**: チャットの揮発的なメモリではなく、後から `/add-feature` で参照できる固定ドキュメントに残す
3. **ステップ毎の依存関係を明示する**: 前→後の一方向依存を明文化し、ロールバック戦略も記載
4. **`/add-feature` の引数を提供する**: そのままコピペで実行できるコマンド文字列を生成

## 使用タイミング

以下のタイミングで使用してください:

- 新機能の構想がまとまり、実装フェーズに入る直前
- 既存機能の大規模拡張（複数ファイル・複数 PR にまたがる）
- 「ステップ分割して」「コマンド化して」「PR 分けたい」と依頼された時
- Plan モードで設計した内容を後で参照可能な形で残したい時

## 実行フロー

### ステップ 1: 既存ドキュメントの確認

`docs/ideas/` 配下の既存ファイルを確認し、命名規則とフォーマットを把握:

```bash
ls docs/ideas/
```

**典型的な既存ファイル**:

- `[feature-name]-plan.md` — 計画書本体
- `[feature-name]-add-feature-commands.md` — `/add-feature` コマンド一覧

既存の同名機能拡張系がある場合は、フォーマットを揃える。

### ステップ 2: コードベース調査（並列）

Explore agent を **最大 3 並列** で投入して以下を把握:

1. **既存の関連実装**: 既存の同種機能（似た拡張・先行 PR）
2. **再利用可能な資産**: ユーティリティ・型・ストア・API ルート・パッケージ依存
3. **設計ドキュメント**: `.steering/` 配下、`docs/` 配下、関連 issue

各 agent には「調査範囲」を明示し、`very thorough` で報告させる。

### ステップ 3: 要件明確化（AskUserQuestion）

設計を固めるために必要な分岐点をユーザに確認。**最大 4 問** までで以下を中心に聞く:

- **機能の範囲**: MVP に含むもの / Post-MVP に回すもの
- **技術選択**: 既存依存だけで足りるか、新規パッケージを追加するか
- **永続化先**: LocalStorage / IndexedDB / セッション中のみ / なし
- **段階分割の希望**: ステップ数の希望、まとめたいか分けたいか
- **対応形式・件数**: ファイル形式、複数 OK か単数か、上限値

新規パッケージ追加や破壊的変更が含まれる場合は、**必ず明示的承認を取る**。

### ステップ 4: 詳細設計（Plan agent）

Phase 1 の調査結果と Phase 3 の回答を踏まえ、Plan agent に以下を依頼:

- データモデル拡張（型定義 + 上限定数）
- プロンプト/ロジック構成（既存関数の拡張点）
- 新規ファイル一覧（パス + 役割）
- UI 拡張案
- ランタイム統合
- 永続化戦略（partialize 設定）
- 段階分割案（ステップ数 + 各ステップの完了範囲）
- テスト計画（ユニット / 統合 / 手動）
- 検証方法（自動ゲート + 手動シナリオ）
- リスク表

### ステップ 5: 計画書ドキュメント保存

`templates/plan.md` を読み、プレースホルダーを置換して保存:

**出力先**: `docs/ideas/[feature-name]-plan.md`

**含めるべきセクション**:

- Context（背景・追加機能の概要・要件確認結果・段階分割方針）
- 設計サマリ（データモデル・ロジック・新規ファイル・UI・ランタイム・永続化・i18n）
- 段階分割（ステップ数 + 各ステップの実装内容）
- 重要な制約・リスク（表）
- Critical Files（既存修正 + 新規作成）
- Verification（自動 + 手動 + ステアリング運用）

### ステップ 6: コマンド一覧ドキュメント保存

`templates/add-feature-commands.md` を読み、各ステップ毎の `/add-feature` コマンドを生成して保存:

**出力先**: `docs/ideas/[feature-name]-add-feature-commands.md`

**含めるべきセクション**:

- 実行順の全体像（コードブロックでフロー図）
- 各ステップ:
  - コマンド文字列（コピペ可能、参照ドキュメント指定込み）
  - 実装内容（ファイル単位の詳細）
  - 動作確認手順
  - 依存（前のステップ）
- 各ステップ完了時点で何が動くか（表）
- ロールバック戦略
- 事前確認事項
- v2 以降で検討する機能

### ステップ 7: ユーザに完了報告

保存した 2 つのドキュメントのパスを提示し、Step 1 着手を打診:

```
作成しました:
- docs/ideas/[feature-name]-plan.md
- docs/ideas/[feature-name]-add-feature-commands.md

Step 1 から着手しますか？
```

## コマンド文字列のフォーマット

各 `/add-feature` コマンドは以下の形式で生成:

```
/add-feature [機能名] [ステップ名]: [実装する範囲の要約 1〜2 文]。参照ドキュメント: docs/ideas/[feature-name]-plan.md (Step N 範囲のみ実装[、Step 1〜N-1 完了前提])
```

**ポイント**:

- 必ず参照ドキュメントを明示し、「該当ステップ範囲のみ実装」を強調
- 依存ステップがある場合は「Step 1 完了前提」「Step 1-2 完了前提」を明記
- そのままコピペで実行できる粒度に整える

## 重要な原則

### MUST（必須）

- Phase 1 で必ず Explore agent を使ってコードベース調査
- AskUserQuestion で要件分岐点を確認（推測で進めない）
- 新規パッケージ依存は **明示的承認** を取る
- 2 ドキュメント（plan + commands）両方を必ず保存
- 既存 `docs/ideas/` のファイルとフォーマットを揃える
- 各 `/add-feature` コマンドに参照ドキュメント指定を入れる
- 段階分割の依存関係を一方向にする

### NEVER（禁止）

- Plan agent の出力をそのままユーザに提示して終わり（必ずドキュメント化する）
- 計画書だけ作ってコマンド一覧を省略
- コマンド一覧だけ作って計画書を省略（参照先がなくなる）
- ステップ間で循環依存を設ける
- 「全部 1 PR でやる」を推奨する（レビュー負荷・切り戻し範囲の観点で避ける）
- 既存パッケージ依存だけで実現可能な場合に新規依存を追加する
- ユーザが希望していないのに 5 ステップ以上に細分化する

## 段階分割の指針

### ステップ数の目安

| 機能規模                          | 推奨ステップ数                            | 例                                              |
| --------------------------------- | ----------------------------------------- | ----------------------------------------------- |
| 小（1〜2 ファイル変更）           | **このスキルを使わず直接 `/add-feature`** | 設定項目 1 つ追加、バグ修正                     |
| 中（3〜5 ファイル、UI＋ロジック） | 1〜2 ステップ                             | 新しい設定タブ 1 個                             |
| 大（型 + ロジック + UI + テスト） | 3〜4 ステップ                             | 新機能の追加（本スキルの本領）                  |
| 超大（複数機能を含む）            | 5〜7 ステップ                             | 面接モード本体（既存の interview-mode-plan.md） |

### 分割の切り口

**良い切り口**:

- 型定義 / 純粋ロジック / UI / ランタイム統合 — レイヤ別
- 機能 A / 機能 B / 統合 — 機能別
- 基盤 / 拡張 / 最適化 — 段階別

**悪い切り口**:

- ファイル単位（依存関係が複雑になる）
- 「型 + ロジック + UI + テスト + ドキュメント」を 1 ステップに詰め込む

## トラブルシューティング

### Plan agent の出力が冗長すぎる

→ ドキュメント保存時に「設計サマリ」のみ抽出。「議論した代替案」「考慮した別アプローチ」は plan ファイルから外す（採用案のみ記載）

### ユーザが「1 PR でやる」と言う

→ 1 ステップ案で進めるが、計画書には「1 ステップ実装」と明記。`/add-feature` コマンドも 1 つだけ生成

### 既存の同種計画書がある

→ 命名・フォーマット・章立てを **完全に揃える**。例えば `interview-mode-plan.md` がある場合、新計画書も `interview-mode-uploads-plan.md` のように接頭辞・接尾辞を踏襲

### Phase 1 で不確定要素が多すぎる

→ AskUserQuestion を **Phase 1 直後にも追加** で発動。要件分岐の確認 → 再 Explore で方向性を絞る
